# 前端对接指南：多轮对话功能

> 写给负责前端的队友。后端已改好，这份文档说明前端需要怎么改才能配合。

---

## 一、改了什么？为什么要改？

**之前的流程：** 用户提问 → AI 一次性返回全部分析 → 展示 5 张卡片，结束。

**现在的流程：** 用户提问 → AI 可能追问 → 用户回答 → AI 再追问或更新分析 → 多轮后才给出建议。

**目的：** 让 AI 在信息不足时主动追问，而不是用模糊信息凑一个敷衍的回答。

---

## 二、后端 API 变化

### 新增了两个接口

| 接口 | 方法 | 用途 |
|------|------|------|
| `/api/analysis/start` | POST | **开始新分析**（会自动清空之前的对话记忆） |
| `/api/analysis/reply` | POST | **用户回答追问后**继续对话 |

旧的 `/api/chat` 接口还在，没有删掉，但新流程不用它了。

### 接口请求格式

**开始分析：**

```javascript
// POST /api/analysis/start
fetch('/api/analysis/start', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    question: "用户输入的问题内容"
  })
})
```

**回复追问：**

```javascript
// POST /api/analysis/reply
fetch('/api/analysis/reply', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    message: "用户的回答内容",
    prompt_name: "mind_lens",
    stream: false
  })
})
```

### 接口返回格式

两个接口返回格式一样，**直接返回 JSON 对象**（后端已经帮你解析好了，不需要前端再 `JSON.parse`）：

```json
{
  "cards": {
    "surface": "表层问题的分析文本（可能是null）",
    "assumption": "隐含假设的分析文本（可能是null）",
    "info_gap": "信息缺失的分析文本（可能是null）",
    "blind_spot": "认知盲点的分析文本（可能是null）",
    "suggestion": "建议文本（大部分情况是null，只有可以给建议时才有值）"
  },
  "follow_up": "AI 的追问问题（如果不需要追问则是null）",
  "ready_for_suggestion": false,
  "round": 1
}
```

如果 LLM 返回了格式不正确的内容，后端会返回一个带 `error` 字段的对象：

```json
{
  "cards": {},
  "follow_up": null,
  "ready_for_suggestion": false,
  "round": 0,
  "error": "LLM 返回了非法 JSON",
  "raw": "原始返回内容的前500字符"
}
```

前端只需要检查是否有 `error` 字段即可做错误处理。

## 四、具体怎么改 result.html

### 4.1 页面结构

需要把现在的页面改成三个区域：

```
┌────────────────────────────────────────┐
│  顶部：5 张卡片（横排或网格）            │
│  [表层问题] [隐含假设] [信息缺失]        │
│  [认知盲点] [建议🔒]                     │
├────────────────────────────────────────┤
│  中间：对话区（聊天记录）                │
│  用户: "我不知道要不要跳槽..."           │
│  AI: "你现在最不满意的是什么？"           │
│  用户: "主要是薪资..."                   │
├────────────────────────────────────────┤
│  底部：输入框 + 发送按钮                 │
└────────────────────────────────────────┘
```

### 4.2 5 张卡片的行为

- 前 4 张卡片（表层问题、隐含假设、信息缺失、认知盲点）：
  - 初始状态显示"等待分析..."
  - 每次收到 AI 回复后，把对应字段的内容更新上去
  - 如果某个字段是 `null`，保持之前的内容不变

- 第 5 张卡片（建议）：
  - 初始状态**锁定**（可以加个锁图标，或者加半透明遮罩）
  - 只有当 `ready_for_suggestion` 为 `true` 且 `cards.suggestion` 有值时才解锁显示

### 4.3 交互流程（JavaScript 逻辑）

```
页面加载
  ↓
从 URL 获取 question 参数
  ↓
调用 POST /api/analysis/start（传 question）
  ↓
解析返回的 JSON
  ↓
更新卡片内容
  ↓
判断 follow_up 是否有值？
  ├── 有值 → 在对话区显示 AI 的追问 → 等用户输入回答
  │           ↓
  │         用户输入后，调用 POST /api/analysis/reply（传用户的回答）
  │           ↓
  │         解析返回的 JSON → 更新卡片 → 再判断 follow_up... 循环
  │
  └── 无值 → 判断 ready_for_suggestion？
      ├── true → 解锁建议卡片，分析完成
      └── false → 分析完成（没有追问也没有建议的情况，一般不会出现）
```

### 4.4 参考代码片段

下面是核心逻辑的参考代码（不是最终代码，只是帮你理解思路）：

```javascript
let currentRound = 0;

// 更新卡片内容
function updateCards(cards) {
  if (cards.surface) {
    document.querySelector('#card-surface .content').textContent = cards.surface;
  }
  if (cards.assumption) {
    document.querySelector('#card-assumption .content').textContent = cards.assumption;
  }
  if (cards.info_gap) {
    document.querySelector('#card-info-gap .content').textContent = cards.info_gap;
  }
  if (cards.blind_spot) {
    document.querySelector('#card-blind-spot .content').textContent = cards.blind_spot;
  }
}

// 解锁建议卡片
function unlockSuggestion(text) {
  const card = document.querySelector('#card-suggestion');
  card.classList.remove('locked');      // 去掉锁定样式
  card.querySelector('.content').textContent = text;
}

// 在对话区添加一条消息
function addMessage(text, sender) {
  // sender: 'user' 或 'ai' 或 'system'
  const div = document.createElement('div');
  div.className = `message ${sender}`;
  div.textContent = text;
  document.querySelector('#chat-messages').appendChild(div);
}

// 处理 AI 返回的数据
function handleResponse(data) {
  currentRound = data.round;
  
  // 1. 更新卡片
  updateCards(data.cards);
  
  // 2. 判断是否有追问
  if (data.follow_up) {
    addMessage(data.follow_up, 'ai');
    // 显示输入框，让用户回答
    showInputArea();
  }
  
  // 3. 判断是否可以显示建议
  if (data.ready_for_suggestion && data.cards.suggestion) {
    unlockSuggestion(data.cards.suggestion);
  }
}

// 开始分析
async function startAnalysis(question) {
  addMessage(question, 'user');
  
  const res = await fetch('/api/analysis/start', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ question })
  });
  const data = await res.json();  // 后端已解析好 JSON，直接用
  handleResponse(data);
}

// 用户回复追问
async function sendReply(message) {
  addMessage(message, 'user');
  
  const res = await fetch('/api/analysis/reply', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ message, prompt_name: 'mind_lens', stream: false })
  });
  const data = await res.json();  // 后端已解析好 JSON，直接用
  handleResponse(data);
}
```

## 五、需要注意的点

### 5.1 追问轮次

追问轮次没有硬性上限，由 LLM 自主判断何时给出建议（一般 2-4 轮即可）。用户也可随时点击"直接给建议"按钮跳过追问。

前端可以显示当前对话轮次（如"第 2 轮"），让用户知道进度。

### 5.2 错误处理

如果 LLM 返回了格式不对的内容，后端会返回带 `error` 字段的对象。前端检查即可：

```javascript
const data = await res.json();
if (data.error) {
  addMessage('分析出现异常：' + data.error, 'system');
  return;
}
handleResponse(data);
```

### 5.3 加载状态

每次调用 API 的时候（等 AI 回复），给个加载提示，比如：
- 发送按钮变灰并显示"分析中..."
- 输入框禁用
- 等回复后恢复

### 5.4 移动端

问卷数据显示 81% 用户来自手机。5 张卡片横排在手机上太挤，建议：
- 手机上改成 2 列或 1 列
- 用 CSS media query：`@media (max-width: 768px) { ... }`

---

## 六、一句话总结改动

> 前端从"调一次 API → 一次性展示"改成"调多次 API → 每次更新卡片 + 显示追问"。核心就是把 `parseAIResponse`（解析纯文本）换成 `JSON.parse`（解析 JSON），然后加一个聊天输入区让用户回答追问。
